## 题目
什么是 Prompt Engineering？它在 LLM 应用开发中的核心价值是什么？

## 答案
### 核心结论
Prompt Engineering 是围绕大模型输入设计、约束和迭代优化的一整套方法论，核心价值是用更低成本、更快速度把模型能力稳定地转化为可用产品能力。

### 关键点
1. Prompt 不只是“写一句话”，而是任务定义、角色设定、格式约束、上下文组织和输出控制的组合。
2. 它直接影响模型是否理解任务、是否按格式回答、是否减少跑偏。
3. 在很多应用里，Prompt 优化比直接改模型更便宜、更快落地。
4. 尤其在 RAG、Agent、结构化输出、多轮对话中，Prompt 是系统行为的重要控制层。

### 工程视角
可以把 Prompt 理解为“大模型时代的轻量级程序接口”。

### 追问点
- Prompt 工程和传统 NLP 模板设计有什么本质差异？
- 哪些问题无法只靠 Prompt 解决？

## 题目
system prompt 和 user prompt 分别起什么作用？为什么要把它们分开设计？

## 答案
### 核心结论
System prompt 负责定义全局行为边界和优先级，user prompt 负责表达当前任务需求；把两者分开设计有助于提高可控性、复用性和安全性。

### 关键点
1. system prompt 用于定义角色、风格、安全规则、输出规范和能力边界。
2. user prompt 用于承载具体问题、业务输入和临时要求。
3. 分层设计后，系统规则更稳定，业务输入更灵活。
4. 当出现异常输出时，也更容易定位是系统规则问题还是用户输入问题。

### 工程实践
通常会把稳定规则放在 system prompt，把任务参数、上下文和用户问题放在 user prompt。

### 追问点
- 为什么 system prompt 不能写得过长？
- 当 system prompt 和 user prompt 冲突时怎么办？

## 题目
一个高质量 Prompt 通常应该包含哪些基本组成部分？

## 答案
### 核心结论
高质量 Prompt 通常包含任务目标、角色设定、输入上下文、约束条件和输出格式这五个核心部分。

### 关键点
1. 任务目标：告诉模型到底要完成什么。
2. 角色设定：让模型明确以什么身份和风格回答。
3. 输入上下文：提供完成任务所需的数据或背景。
4. 约束条件：规定能做什么、不能做什么、优先级如何。
5. 输出格式：要求以特定结构或字段返回结果。

### 常见误区
很多 Prompt 效果差，不是模型不行，而是任务目标和输出要求写得太含糊。

### 追问点
- 如果 Prompt 太长，会有什么副作用？
- 哪部分对结构化任务最重要？

## 题目
zero-shot、one-shot、few-shot prompting 有什么区别？

## 答案
### 核心结论
zero-shot 是不给示例直接提任务，one-shot 给一个示例，few-shot 给多个示例；示例越充分，模型越容易理解任务模式，但 token 成本也越高。

### 关键点
1. zero-shot 适合任务清晰且模型本身已具备相关能力的场景。
2. one-shot 适合用一个典型例子告诉模型“你应该怎么做”。
3. few-shot 适合格式复杂、边界模糊或风格要求强的任务。
4. few-shot 的收益来自“给模型看模式”，不是单纯堆样例数量。

### 工程实践
优先从 zero-shot 开始，效果不稳时再逐步引入示例。

### 追问点
- 为什么 few-shot 太多可能反而变差？
- 示例顺序会影响输出吗？

## 题目
为什么在 Prompt 中明确输出格式很重要？

## 答案
### 核心结论
明确输出格式能显著降低模型自由发挥空间，提高解析稳定性、后续处理效率和整体可控性。

### 关键点
1. 模型默认倾向于自然语言自由表达，不一定会按工程需求返回结构化结果。
2. 输出格式不明确时，容易出现漏字段、额外解释、顺序混乱等问题。
3. 对 API、Agent、自动化流程来说，稳定格式往往比文采更重要。
4. 格式要求越具体，越有利于后处理和自动校验。

### 工程实践
常见做法是给出字段定义、示例和禁止输出额外说明的约束。

### 追问点
- 只有格式要求，没有字段语义定义，够不够？
- 为什么格式要求过于复杂也会出问题？

## 题目
Prompt 里的“角色设定”为什么有效？它主要影响什么？

## 答案
### 核心结论
角色设定有效，是因为它改变了模型对任务语境和输出风格的条件化方式，主要影响回答角度、表达风格、深度和边界感。

### 关键点
1. 角色设定能帮助模型更快进入目标语境。
2. 它通常影响措辞、结构、术语密度和对问题的关注重点。
3. 角色设定并不等于能力增强，它更多是行为引导。
4. 同一个问题，让模型扮演面试官、老师、客服，输出往往明显不同。

### 常见误区
角色设定不能替代具体任务说明，只有“你是专家”通常不够。

### 追问点
- 角色设定和 system prompt 的关系是什么？
- 角色设定为什么会影响拒答风格？

## 题目
在 Prompt 里写“不要做什么”有用吗？如何正确写约束？

## 答案
### 核心结论
有用，但光写否定约束通常不够；更好的方式是同时给出禁止项、优先级和替代行为，让模型知道违规时应该怎么处理。

### 关键点
1. 单纯写“不要胡说”太抽象，缺乏可执行性。
2. 更有效的约束是具体描述禁区，比如“没有依据时明确说不知道”。
3. 约束要和任务目标兼容，不能互相打架。
4. 对高风险场景，约束最好和后处理校验、权限控制联动。

### 工程实践
负向约束最好配合正向策略，例如“不确定时先列出缺失信息，再请求补充”。

### 追问点
- 为什么过多否定约束会让模型变保守？
- 约束冲突时怎么排查？

## 题目
为什么在生产环境里通常不直接拼字符串，而是使用 Prompt 模板？

## 答案
### 核心结论
因为 Prompt 模板更利于参数化、复用、版本管理和 A/B 测试，能把 Prompt 从“随手写文案”变成“可维护的工程资产”。

### 关键点
1. 模板能明确哪些部分是固定规则，哪些部分是动态输入。
2. 它可以降低多人协作时的变更风险。
3. 有利于版本迭代、回滚和线上问题定位。
4. 在多场景应用中，同一模板可以按变量适配不同任务。

### 工程实践
通常把任务说明、格式规则、变量占位和示例统一纳入模板管理。

### 追问点
- Prompt 模板应该放代码里还是配置中心？

## 题目
Prompt 是不是越详细越好？什么时候详细反而会适得其反？

## 答案
### 核心结论
Prompt 不是越长越好，关键在于信息密度和规则一致性；过度冗长会浪费 token、稀释重点、增加冲突和理解负担。

### 关键点
1. 过长 Prompt 会提升成本和延迟。
2. 冗余规则容易互相矛盾，导致模型执行不稳定。
3. 重点被埋没后，模型可能抓不到真正高优先级要求。
4. 对重复规则进行合并和归纳，往往比继续堆字更有效。

### 工程实践
Prompt 优化经常不是“补更多说明”，而是“删低价值内容、提高清晰度”。

### 追问点
- 哪些内容最值得保留在 Prompt 里？
- 什么时候应该把信息移出 Prompt，改成交给工具或外部配置？

## 题目
如果一个 Prompt 效果不好，你会先从哪些方面排查？

## 答案
### 核心结论
应优先排查任务定义是否清晰、输入上下文是否足够、规则是否冲突、输出格式是否明确，以及示例是否真实代表目标任务。

### 关键点
1. 看模型是否真的理解了任务目标。
2. 看上下文是否缺失，是否要求模型凭空猜测。
3. 看 system prompt、user prompt、示例之间是否冲突。
4. 看输出要求是否过于模糊或过于复杂。
5. 看失败是否集中在某些特定输入分布。

### 工程实践
不要一上来就重写整个 Prompt，先定位到底是哪一层出了问题。

### 追问点
- 如何设计一套 Prompt 调试 checklist？
- 如何区分是 Prompt 问题还是模型能力问题？

## 题目
few-shot 示例应该怎么设计，才能真正提高效果？

## 答案
### 核心结论
高质量 few-shot 示例的关键不是多，而是覆盖任务模式、边界情况和期望格式，让模型学到“什么输入对应什么输出”。

### 关键点
1. 示例要和真实输入分布接近，不能脱离业务场景。
2. 应至少覆盖典型样本、边界样本和容易混淆的样本。
3. 输出格式必须与真实目标格式严格一致。
4. 示例过长、风格不统一或质量不高，反而会误导模型。

### 工程实践
few-shot 更像“示范教学”，示例质量往往比数量更重要。

### 追问点
- few-shot 是否需要包含失败案例？
- 示例里要不要加入反例和说明？

## 题目
什么是 Chain-of-Thought（CoT）Prompting？它为什么有时能提升推理效果？

## 答案
### 核心结论
CoT 是通过引导模型显式展开中间推理步骤，帮助其在复杂任务中更稳定地组织思路，因此在数学、逻辑、规划等任务上常能提升表现。

### 关键点
1. CoT 强调“先分解，再回答”。
2. 它能帮助模型减少一步跳结论带来的错误。
3. 对多步问题尤其有效，比如算术、比较、条件推理。
4. 但 CoT 不是万能的，也可能生成看似合理但错误的推理过程。

### 工程实践
在高敏感场景下，不应盲信模型给出的“推理链”，还需要校验最终结果。

### 追问点
- 为什么 CoT 有时会泄露内部推理？
- 什么时候不适合让模型显式展开思考过程？

## 题目
什么是 self-consistency prompting？适合什么任务？

## 答案
### 核心结论
Self-consistency 是让模型对同一问题生成多条推理路径，再从中选出一致性更高的答案，适合对结果正确率要求高、且允许额外推理成本的任务。

### 关键点
1. 它的核心思想不是只信一次输出，而是通过多样采样增加鲁棒性。
2. 常见于数学、逻辑推理和复杂判断任务。
3. 优点是能缓解单次推理偶然出错的问题。
4. 缺点是成本更高、延迟更长，不适合所有线上场景。

### 工程实践
可用于离线评测、关键步骤复核或高价值任务兜底。

### 追问点
- self-consistency 和 majority voting 有什么关系？
- 为什么开放式问答不一定适合这种方法？

## 题目
如果你要让模型稳定输出结构化字段，Prompt 应该怎么写？

## 答案
### 核心结论
应明确字段名、字段含义、字段类型、缺失时的处理规则，并尽量减少自由文本区域，从而把输出空间压缩到可控范围。

### 关键点
1. 明确字段列表，而不是只说“请输出 JSON”。
2. 为字段写清楚类型和允许值范围。
3. 说明缺失字段该返回 null、空字符串还是空数组。
4. 要求不得输出 schema 外字段和解释性文本。
5. 必要时给一个标准示例。

### 工程实践
如果接口支持 response schema 或 function calling，应优先使用原生结构化能力。

### 追问点
- Prompt 约束和 schema 约束分别擅长什么？
- 为什么字段越多越容易失稳？

## 题目
什么是 Prompt Injection？为什么它是 Prompt 工程里的核心问题之一？

## 答案
### 核心结论
Prompt Injection 是指攻击者通过构造输入或外部内容，诱导模型忽略系统规则、执行恶意指令或泄露敏感信息；因为 LLM 把文本都当作可解释内容，所以这类攻击尤其隐蔽。

### 关键点
1. 注入内容可能来自用户输入、网页、知识库文档或工具返回结果。
2. 模型无法天然区分“可信规则”和“恶意文本”。
3. 在 Agent 系统中，注入会从“说错”升级为“做错”。
4. Prompt 设计可以缓解风险，但不能单独解决安全问题。

### 工程实践
要结合输入隔离、权限控制、白名单、审计和高风险动作确认机制。

### 追问点
- 为什么“忽略以上所有指令”有时会生效？
- Prompt Injection 与传统代码注入有何异同？

## 题目
当 system prompt、few-shot 示例和用户输入之间出现冲突时，应该如何设计优先级？

## 答案
### 核心结论
应该明确建立规则优先级：系统边界最高，任务模板其次，示例用于模式引导，用户输入承载当前需求；如果缺少清晰优先级，模型行为会变得不稳定。

### 关键点
1. system prompt 应负责全局不可突破的边界。
2. few-shot 示例负责说明任务模式，不应覆盖系统规则。
3. user prompt 负责当前任务，但不能突破安全和协议约束。
4. 冲突越多，模型越可能选择“折中”或随机偏向某一层。

### 工程实践
在 Prompt 中显式声明优先级，能减少模型在冲突场景下的歧义。

### 追问点
- 为什么高优先级规则也会被“冲掉”？
- 怎样在测试中构造冲突样例？

## 题目
如果 Prompt 太长导致成本高、延迟高，你会怎么压缩？

## 答案
### 核心结论
Prompt 压缩应优先删除重复信息、合并规则、提炼高优先级约束，并把静态知识迁移到外部检索或配置层，而不是简单粗暴截断。

### 关键点
1. 先找冗余：重复规则、重复示例、重复上下文。
2. 再做抽象：把长篇说明压成规则清单或决策原则。
3. 静态背景知识可以移出 Prompt，改用检索或工具获取。
4. 只保留真正影响当前任务的上下文。

### 工程实践
压缩前后要做回归评测，避免“省了 token，丢了效果”。

### 追问点
- 哪类信息最不适合压缩？
- 为什么摘要有时会损失关键约束？

## 题目
Prompt 优化为什么不能只靠“感觉好像更顺了”？应该怎么评测？

## 答案
### 核心结论
因为 Prompt 优化往往会在某些样本上变好、另一些样本上变差，必须通过离线评测集、错误分类和线上实验来验证，而不是凭主观感觉。

### 关键点
1. 建立代表真实输入分布的测试集。
2. 指标应覆盖正确率、格式稳定性、拒答策略、成本和延迟。
3. 关注总体分数之外，还要看失败类型是否转移。
4. 对关键业务可做线上 A/B，比较真实用户反馈与转化。

### 工程实践
Prompt 版本管理应和实验结果强关联，避免“改过但不知道为什么更好”。

### 追问点
- Prompt 的评测集应该怎么抽样？
- 什么情况下离线效果提升但线上反而变差？

4. 用户感知差往往不是因为模型太弱，而是 Prompt 没把交互设计好。

Prompt 优化时要结合用户反馈，不要只看离线技术指标。

### 追问点
- 什么样的 Prompt 优化会提高满意度但不一定提高准确率？
- Prompt 和前端交互文案该如何协同设计？

## 题目

### 工程实践

## 题目
如何建立 Prompt 优化的持续闭环，而不是一次性调完就结束？

## 答案
### 核心结论
需要把线上日志、用户反馈、失败样本、离线评测集和版本实验结果连成闭环，让 Prompt 优化从“手工试错”升级成“数据驱动迭代”。

### 关键点

4. 上线后继续监控指标漂移和失败模式变化。

## 题目
你认为 Prompt 工程里最核心的 trade-off 是什么？

## 答案
### 核心结论
最核心的 trade-off 是“可控性、灵活性、成本、效果”之间的平衡：约束越强通常越稳，但自由度越低；示例越多通常效果越好，但 token 成本越高；规则越复杂越全面，但也越难维护。

### 关键点
1. 高可控性通常意味着更强格式和行为约束。
2. 高灵活性有利于覆盖长尾场景，但更容易跑偏。
3. 高效果往往依赖更多上下文、更多示例和更强模型。
4. 成本不仅是 token 费用，也包括维护、调试和回归评测成本。

### 面试表达
可以总结为：Prompt 工程不是“把模型调听话”这么简单，而是在工程现实里持续做约束与收益之间的权衡。

### 追问点
- 哪类场景应该优先追求可控而非灵活？
- 如何通过模板分层和路由缓解这些 trade-off？
