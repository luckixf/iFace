## 题目
什么是大语言模型（LLM）？它和传统机器学习模型的核心区别是什么？

## 答案
### 核心结论
LLM 是一种基于海量语料预训练、能够进行通用语言理解与生成的模型；与传统机器学习模型相比，它更强调大规模预训练后的泛化能力，而不是单一任务上的定制建模。

### 关键点
1. LLM 通常基于 Transformer 架构，训练目标常见为 next token prediction。
2. 传统机器学习模型通常针对单一任务训练，如分类、回归、排序。
3. LLM 先做通用预训练，再通过指令微调、对齐训练等方式适配多任务。
4. LLM 的能力边界受模型规模、训练数据、上下文窗口和对齐方式共同影响。

### 面试表达
可以概括为：传统模型像“专项选手”，LLM 更像“通才底座 + 对话接口”。

### 追问点
- 为什么只做“预测下一个 token”也能学出复杂能力？
- 为什么 LLM 会出现幻觉？

## 题目
什么是 token？token 与汉字、英文单词之间是什么关系？

## 答案
### 核心结论
token 是模型处理文本时的最小离散单位，它既不等于“一个字”，也不等于“一个单词”，而是 tokenizer 按规则切分出的片段。

### 关键点
1. 中文中一个字常常接近一个 token，但并不绝对。
2. 英文中一个单词可能对应一个 token，也可能被拆成多个子词。
3. LLM 的输入、输出、计费、上下文长度，本质上都以 token 为单位。
4. Prompt、历史消息、RAG 检索结果和工具输出都会共同消耗 token 预算。

### 工程意义
如果 token 使用不受控，就会带来上下文溢出、延迟升高和成本增加。

### 追问点
- 为什么不同模型的 tokenizer 不一样？
- 为什么相同一句话在不同模型上的 token 数会差很多？

## 题目
Tokenizer 的作用是什么？常见的 tokenizer 思路有哪些？

## 答案
### 核心结论
Tokenizer 的作用是把原始文本转换为模型可处理的 token 序列，并在生成后把 token 反解成文本；它直接影响上下文利用率、成本和模型对文本边界的理解方式。

### 关键点
1. 模型不能直接处理字符串，必须先把文本映射成 token id。
2. 常见 tokenizer 思路包括 BPE、WordPiece、SentencePiece 等。
3. 好的 tokenizer 需要在词汇表大小、切分粒度和泛化能力之间做平衡。
4. tokenizer 不同，会影响长度估计、提示词设计和多语言表现。

### 工程实践
做上下文预算时，不能凭字符数估算，最好按目标模型真实 tokenizer 统计。

### 追问点
- tokenizer 为什么会影响代码场景表现？
- 多语言任务中 tokenizer 设计有哪些权衡？

## 题目
什么是上下文窗口（context window）？为什么它对 LLM 应用很重要？

## 答案
### 核心结论
上下文窗口是模型单次推理时可见的总 token 上限，它决定模型一次能参考多少历史信息、检索内容和工具结果。

### 关键点
1. 上下文窗口包含 system prompt、user 输入、历史消息、检索内容、工具输出和模型待生成空间。
2. 超出窗口后，内容会被截断，模型就看不到被裁掉的信息。
3. 多轮对话、长文总结、RAG 和 agent 调用都会竞争这部分预算。
4. 窗口越大不代表效果一定越好，长上下文可能带来注意力稀释和成本上升。

### 工程实践
常见手段包括消息裁剪、摘要记忆、检索压缩、只保留高价值上下文。

### 追问点
- 为什么 128k 上下文不等于 128k 都能高质量利用？
- 为什么长上下文会显著增加延迟？

## 题目
Transformer 为什么会成为大语言模型的主流架构？

## 答案
### 核心结论
Transformer 能成为主流，是因为它依赖自注意力机制，擅长建模长距离依赖，并且相比 RNN 更容易并行训练，适合大规模预训练。

### 关键点
1. 自注意力允许每个 token 直接关注序列中其他位置的信息。
2. Transformer 的训练并行度高，适合在海量数据和大规模算力上扩展。
3. 相比 RNN，Transformer 更擅长处理长依赖问题。
4. 扩展规律更稳定，适合“更大模型 + 更多数据 + 更长训练”的路线。

### 面试表达
可以说：Transformer 不是因为“最聪明”才赢，而是因为它在效果、并行训练和扩展性上综合最优。

### 追问点
- 自注意力的计算瓶颈是什么？
- 为什么长上下文下 attention 成本会变高？

## 题目
请解释一下 Transformer 里的 self-attention 机制，以及 Q、K、V 分别代表什么。

## 答案
### 核心结论
Self-attention 的核心是：每个 token 根据自身查询向量 Q，与其他 token 的键向量 K 计算相关性，再对对应的值向量 V 加权求和，得到新的表示。

### 关键点
1. Q 表示“我想找什么信息”。
2. K 表示“我这里有什么信息可以被匹配”。
3. V 表示“真正被取出的内容”。
4. attention 分数通常由 Q 和 K 的相似度得到，再经 softmax 归一化。
5. 最终结果是对多个 V 的加权和。

### 直观理解
可以把它理解成：每个 token 都在对全局做一次“检索 + 聚合”。

### 追问点
- 为什么要缩放点积？
- 多头注意力相比单头注意力的优势是什么？

## 题目
一个通用大模型通常会经历哪些训练阶段？

## 答案
### 核心结论
通用大模型的典型训练流程是：预训练打基础，监督微调让模型学会按指令回答，对齐训练让输出更符合人类偏好和安全约束。

### 关键点
1. 预训练：在大规模语料上做语言建模，获得基础知识和语言能力。
2. SFT：用高质量指令数据微调，让模型更会“听指令”。
3. 对齐训练：常见有 RLHF、DPO 等，让模型输出更符合人类偏好。
4. 某些模型还会做领域继续预训练或专项蒸馏。

### 工程意义
如果业务场景要求稳定遵循格式、少跑偏，往往不能只依赖预训练模型，需要微调或更强约束。

### 追问点
- 预训练和 SFT 分别改变模型的什么能力？
- RLHF 和 DPO 有什么区别？

## 题目
什么是 system prompt？它在大模型应用里起什么作用？

## 答案
### 核心结论
System prompt 是给模型设定全局行为边界和优先级的控制指令，它通常决定模型的角色、风格、输出格式和安全约束。

### 关键点
1. system prompt 一般优先级高于普通 user prompt。
2. 它用于定义角色、回答边界、禁答规则、格式要求等。
3. 在 agent 场景里，system prompt 往往还规定工具调用规则和决策原则。
4. 它不是绝对安全边界，仍可能被 prompt injection 干扰。

### 工程实践
好的 system prompt 应短而清晰，避免相互冲突和冗长堆砌。

### 追问点
- system prompt 为什么不能完全防 prompt injection？
- 多轮对话里 system prompt 应如何长期保持有效？

## 题目
temperature 和 top_p 分别控制什么？项目里怎么选取合适参数？

## 答案
### 核心结论
temperature 主要控制随机性强弱，top_p 主要控制采样候选范围；任务越偏确定性，参数通常越保守，任务越偏创意生成，参数可适当放开。

### 关键点
1. temperature 越低，输出越稳定、保守；越高，输出越发散。
2. top_p 表示从累计概率达到阈值的候选集合中采样。
3. 两者都影响多样性，但调节机制不同。
4. 抽取、分类、结构化输出等任务通常适合低 temperature。
5. 文案创作、头脑风暴等任务可以适当提高 temperature。

### 常见误区
不是“温度越低越正确”，温度太低也可能导致模式塌缩和表达僵化。

### 追问点
- 在结构化输出里为什么经常把 temperature 设很低？
- 同时调 temperature 和 top_p 时要注意什么？

## 题目
什么是大模型幻觉（hallucination）？它为什么会发生？

## 答案
### 核心结论
幻觉是指模型输出看似合理但实际上不真实、不准确或无法依据上下文验证的内容，其根源在于语言生成目标并不等同于事实校验目标。

### 关键点
1. 模型本质上是在生成“概率上合理”的文本，而不是数据库查询器。
2. 训练语料噪声、知识缺失、上下文不足都会导致幻觉。
3. 提问模糊、约束不清、强制回答也会提高幻觉概率。
4. 长链路 agent 中，检索错误、工具错误、记忆污染也会放大幻觉。

### 工程缓解手段
- RAG 提供外部依据
- 要求引用来源
- 减少开放式猜测
- 对关键任务做规则校验和人工兜底

### 追问点
- 幻觉能彻底消除吗？
- 为什么更大的模型有时看起来更“自信地胡说”？

## 题目
Base model 和 instruction-tuned/chat model 有什么区别？

## 答案
### 核心结论
Base model 更像纯预训练语言模型，擅长续写分布；instruction-tuned 或 chat model 则是在此基础上进一步学会理解人类指令、按要求输出和进行对话。

### 关键点
1. Base model 更接近“文本补全器”，未必擅长遵循复杂任务说明。
2. Chat model 经过 SFT 和对齐，更擅长问答、格式输出和安全响应。
3. 同样的问题，base model 可能延续文本风格，chat model 更可能直接回答。
4. 对业务应用来说，chat model 往往是更实际的选择。

### 工程实践
如果你做面向用户的对话或 agent 系统，通常优先选择 instruction-tuned 模型，而不是裸 base model。

### 追问点
- base model 是否更适合某些蒸馏或领域微调场景？
- chat model 为什么有时更“保守”？

## 题目
zero-shot、one-shot、few-shot prompting 分别是什么意思？什么时候用 few-shot？

## 答案
### 核心结论
zero-shot 是不给示例直接让模型完成任务，one-shot 和 few-shot 是提供一个或多个示例帮助模型理解任务模式；当任务规则抽象、格式要求强或边界容易混淆时，few-shot 往往更有效。

### 关键点
1. zero-shot 成本低，适合通用能力已足够强的任务。
2. one-shot/few-shot 能显著提升格式稳定性和边界一致性。
3. few-shot 对抽取、分类、改写、风格模仿很常见。
4. 代价是会增加 token 消耗，并可能引入示例偏差。

### 工程实践
few-shot 示例应覆盖典型输入、边界输入和反例，且格式要和目标输出一致。

### 追问点
- few-shot 太多为什么可能反而变差？
- 示例顺序会影响结果吗？

## 题目
什么是 embedding？它和生成式大模型是什么关系？

## 答案
### 核心结论
Embedding 是把文本、代码或其他对象映射成稠密向量表示的方法，核心作用是表达语义相似性；它不负责生成内容，但常作为检索、聚类、召回等能力的基础。

### 关键点
1. 相似语义的文本，在向量空间里通常更接近。
2. Embedding 模型主要用于“表示”和“匹配”，不是直接生成答案。
3. 在 RAG 里，embedding 常用于向量检索的召回阶段。
4. 生成式 LLM 和 embedding 模型经常配合使用：前者负责回答，后者负责找资料。

### 工程意义
很多 LLM 应用效果不好，不是生成模型差，而是检索阶段的 embedding 和召回质量不够。

### 追问点
- embedding 检索和关键词检索各自适合什么场景？
- 为什么还需要 rerank？

## 题目
训练（training）和推理（inference）有什么区别？为什么线上成本主要看推理？

## 答案
### 核心结论
训练是让模型学参数，推理是用已训练好的模型对输入生成输出；对绝大多数应用团队来说，线上长期成本和体验主要由推理阶段决定。

### 关键点
1. 训练成本高，但通常由模型提供方承担，业务团队不一定自己做。
2. 推理直接影响用户等待时间、接口成本和并发承载能力。
3. 每次用户请求都会触发推理，因此推理成本会随着调用量线性放大。
4. 推理优化包括模型选型、缓存、裁剪上下文、批处理和量化部署。

### 工程表达
对应用团队来说，“模型是否足够便宜、足够快、足够稳”通常比“训练细节”更重要。

### 追问点
- 为什么输出 token 通常比输入 token 更贵？
- 为什么推理吞吐和首 token 延迟都很关键？

## 题目
在 LLM 应用里，延迟通常由哪些因素组成？首 token 延迟为什么重要？

## 答案
### 核心结论
LLM 延迟通常由网络耗时、排队耗时、Prompt 处理、模型推理和流式输出组成；首 token 延迟决定用户“多久看到系统开始回应”，对体验影响尤其大。

### 关键点
1. 输入越长，prefill 计算越重，首 token 延迟通常越高。
2. 模型越大、并发越高、服务越拥塞，整体推理延迟越高。
3. 工具调用、检索、后处理也会增加端到端时延。
4. 流式输出能改善体感，即使总时长不变。

### 工程优化
- 控制输入长度
- 选更合适的模型
- 使用流式输出
- 做结果缓存和链路并行

### 追问点
- prefill 和 decode 的性能特征有什么不同？
- 为什么短回答任务更关注首 token 延迟？

## 题目
贪心解码、采样解码有什么区别？为什么聊天模型通常不使用 beam search？

## 答案
### 核心结论
贪心解码每步都选概率最高的 token，输出稳定但单一；采样解码能带来多样性，更适合开放式生成；beam search 在聊天场景里成本较高，且未必带来更自然的回答。

### 关键点
1. 贪心解码适合确定性强的任务，如部分结构化生成。
2. 采样解码更适合开放问答、创意生成和自然对话。
3. beam search 会保留多个候选路径，计算成本更高。
4. 对对话模型而言，过强的“最优路径搜索”不一定等价于更自然、符合偏好的输出。

### 工程实践
面向用户的 LLM 接口通常采用温度 + top_p 的采样组合，而不是复杂搜索。

### 追问点
- 什么时候贪心解码反而更合适？
- 采样为什么会带来不稳定性？

## 题目
什么是 scaling law？它为什么对大模型发展很重要？

## 答案
### 核心结论
Scaling law 指模型性能会随着参数规模、数据规模和训练计算量按相对可预测的规律提升，这为“大模型路线”提供了经验依据。

### 关键点
1. 在一定范围内，更多参数、更多数据、更多训练通常能提升能力。
2. 这使得行业敢于投资更大模型，而不是只做架构小修小补。
3. 但扩展并非无限有效，数据质量、训练稳定性和推理成本都会成为瓶颈。
4. 从应用角度看，scale 不只提升能力，也会提升成本和部署复杂度。

### 面试表达
可以说：scaling law 解释了为什么“堆规模”在大模型时代不是简单暴力，而是有统计规律支撑的工程路线。

### 追问点
- 为什么后期数据质量会越来越重要？

## 题目
为什么预训练模型还需要做 instruction tuning？

## 答案
### 核心结论
因为预训练模型学到的是语言分布和知识表达能力，但不一定天然会“按人类想要的方式完成任务”；instruction tuning 的目的是把“会说话”变成“会按要求回答”。

### 关键点

## 题目
请解释一下 LLM 推理里的 prefill 和 decode 阶段，它们在性能特征上有什么区别？

## 答案
### 核心结论
Prefill 是模型先处理全部输入上下文、建立中间状态的阶段；decode 是基于已有状态逐 token 生成输出的阶段。两者瓶颈不同，优化手段也不同。

### 关键点
1. Prefill 的工作量主要受输入 token 长度影响。
2. Decode 是逐 token 生成，输出越长，总时间越长。
3. 长 prompt 往往主要拖慢 prefill，长回答则主要拖慢 decode。
4. KV Cache 的价值在于避免 decode 时重复计算前文。

### 工程意义
如果你的场景输入超长但输出很短，就要优先优化 prefill；如果输出很长，就要关注 decode 吞吐。

### 追问点
- 为什么 KV Cache 对 decode 更重要？
- 流式输出改善的是哪一部分体验？

## 题目
什么是 KV Cache？它为什么能显著提升推理效率？

## 答案
### 核心结论
KV Cache 是把历史 token 对应的 key/value 表示缓存下来，避免模型在逐 token 生成时反复重算历史上下文，从而显著降低 decode 阶段开销。

### 关键点
1. 在自回归生成里，每生成一个新 token 都要参考历史上下文。
2. 如果不缓存，历史部分会重复参与大量计算。
3. 缓存 K/V 后，新 token 只需计算增量部分。
4. 代价是显存占用增加，长上下文和大 batch 下更明显。

### 工程实践
推理服务在高并发场景下，需要在吞吐和显存之间平衡 KV Cache 策略。

### 追问点
- 为什么上下文越长，KV Cache 占用越大？
- prefix cache 和 KV Cache 有什么关系？

4. 这种分工能降低幻觉，提高可控性。

### 工程实践
“模型负责决策，工具负责执行”是很多 agent 系统的基础原则，但前提是权限控制和异常处理做好。

### 追问点
- 哪些场景不值得接工具？
- 工具调用失败后如何兜底？

## 题目
闭源模型和开源模型在企业落地时分别有哪些优缺点？

## 答案
### 核心结论
闭源模型通常在综合能力、服务成熟度和开箱即用性上更有优势；开源模型则在可控性、私有化部署、定制空间和数据安全上更有吸引力。

### 关键点
1. 闭源模型优势：能力强、更新快、运维成本低、生态完善。
2. 闭源模型不足：价格、速率限制、数据合规和可解释性受限。
3. 开源模型优势：可私有部署、可微调、可定制、可离线运行。
4. 开源模型不足：部署复杂、调优成本高、效果未必稳定。

### 工程判断
是否选开源，不只是看模型本身，而是看团队是否有足够的 MLOps 和推理基础设施能力。

### 追问点
- 什么情况下必须选开源自部署？
- 为什么很多团队会采用“闭源兜底 + 开源走量”的混合策略？

## 题目
如果让你评估一个 LLM 是否适合客服问答场景，你会怎么设计评测？

## 答案
### 核心结论
评测不能只看“答得像不像”，而要同时评估正确性、引用依据、拒答策略、语气风格、稳定性、延迟和成本，最好覆盖离线集与线上回放集。

### 关键点
1. 构建真实问题集，覆盖常见问题、边界问题、无答案问题和恶意问题。
2. 评估维度包括：答案正确性、是否引用正确资料、是否胡编、拒答是否得当。
3. 加入格式稳定性、延迟、token 成本和异常率指标。
4. 对关键问题可以做人工评审，对大规模样本可以配合自动判分。

### 工程实践
要特别重视“无答案时能否稳健拒答”，这通常比“能答对简单问题”更体现上线安全性。

### 追问点
- 自动评测和人工评测分别适合什么？

## 题目
多轮对话系统里的“记忆”本质上是什么？和人类记忆有什么不同？

## 答案
### 核心结论
LLM 系统里的“记忆”本质上不是模型像人一样长期记住事情，而是通过上下文重放、摘要、外部存储和检索，把过去信息再次提供给模型使用。

### 关键点
1. 模型参数内知识不是会话级记忆。
2. 会话记忆通常依赖历史消息、摘要记忆、用户画像库或外部数据库。
3. “记住”其实是“在当前推理时再次送入相关信息”。
4. 记忆系统如果设计不好，会引入过期信息、权限泄露和上下文污染。

### 工程实践
多轮系统应区分短期会话记忆和长期用户记忆，并明确何时写入、何时召回、何时遗忘。

### 追问点
- 为什么摘要记忆会丢失细节？
- 长期记忆如何做权限隔离？

## 题目
一个 LLM 应用线上效果突然变差，你会如何排查问题？

## 答案
### 核心结论
排查应沿着“输入变了、模型变了、上下文变了、检索变了、工具变了、输出校验变了”这条链路系统定位，而不是只盯着模型本身。

### 关键点
1. 先看是否有模型版本、参数、系统 prompt 变更。
2. 再看输入分布是否变化，如用户问题类型、语言、长度。
3. 检查检索命中率、工具成功率、超时率和重试情况。
4. 对比历史样本回放，找出失败模式是否集中在某一类场景。
5. 查看是否是缓存污染、上下文截断或限流造成的退化。

### 工程方法
做 LLM Debug 时，保留完整链路日志非常关键，包括 prompt、检索片段、工具调用、模型响应和后处理结果。

### 追问点
- 为什么 LLM 系统的排障比普通接口更难？
- 哪些日志必须采集？

## 题目
设计一个面向企业内部知识问答的 LLM 系统时，你认为最核心的 trade-off 是什么？

## 答案
### 核心结论
最核心的 trade-off 通常是效果、成本、延迟和安全四者之间的平衡：答得更准往往意味着更长上下文、更复杂检索和更贵模型，但这会增加成本和时延，还可能带来更多风险面。

### 关键点
1. 更强模型通常更贵，且不一定在所有任务上都值得。
2. 更长上下文和更多检索片段可能提升覆盖率，但也可能引入噪声。
3. 更激进的自动化能提升效率，但也会放大错误代价。
4. 更严格的安全策略能降低风险，但也可能提高拒答率、影响体验。

### 面试表达
可以说：LLM 系统设计的关键不是把某个指标拉满，而是在业务目标下找到可持续的平衡点。

### 进一步展开
一个成熟系统往往会采用分层策略：简单问题走便宜快模型，复杂问题走强模型；高风险场景加人工审核；关键答案要求引用依据和审计链路。

### 追问点
- 哪类业务更优先考虑安全而不是体验？
- 如何通过路由和缓存缓解这类 trade-off？
