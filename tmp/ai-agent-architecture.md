## 题目
什么是 AI Agent？它和普通聊天机器人最大的区别是什么？

## 答案
### 核心结论
AI Agent 是能够基于目标进行感知、推理、决策并执行动作的智能体；与普通聊天机器人相比，它不仅“会回答”，还能够“会行动、会调用工具、会管理状态”。

### 关键点
1. 普通聊天机器人通常以单轮或多轮对话问答为主。
2. Agent 更强调目标驱动和任务完成，而不仅是语言回复。
3. Agent 往往具备工具调用、状态管理、任务分解和结果验证能力。
4. 聊天机器人可以是 Agent 的一个界面，但不是所有聊天机器人都是 Agent。

### 面试表达
可以概括为：聊天机器人主要解决“说什么”，Agent 进一步解决“怎么做、做完没有、下一步是什么”。

### 追问点
- 什么情况下不需要用 Agent？
- 一个带工具调用的聊天助手，是否已经算 Agent？

## 题目
AI Agent 和普通 LLM 应用的边界在哪里？

## 答案
### 核心结论
普通 LLM 应用通常是“输入一次、输出一次”的增强型推理接口，而 Agent 架构强调持续状态、任务分解、工具协同和闭环执行。

### 关键点
1. 普通 LLM 应用的核心是生成回答。
2. Agent 的核心是围绕目标进行计划、执行、观察和修正。
3. Agent 往往不是单一步骤，而是一条任务链路。
4. 是否需要显式状态和行动闭环，是区分两者的重要标准。

### 工程判断
如果场景只需要问答、总结、抽取，通常不用上 Agent；如果要跨步骤完成任务、调用外部系统、处理失败恢复，才更适合 Agent。

### 追问点
- 带 RAG 的问答系统算不算 Agent？
- Agent 和 workflow 的区别是什么？

## 题目
一个最基础的 Agent 执行循环通常包含哪些步骤？

## 答案
### 核心结论
最基础的 Agent 循环通常可以抽象为：接收目标、分析当前状态、决定下一步动作、执行动作、观察结果、更新状态，并重复直到任务结束。

### 关键点
1. Goal：明确用户目标或系统目标。
2. State：读取当前上下文、历史记录和环境状态。
3. Decide：决定下一步做什么。
4. Act：调用工具、检索、生成内容或触发外部行为。
5. Observe：读取执行结果。
6. Update：更新记忆和状态，判断是否结束。

### 工程意义
这个循环越明确，Agent 越容易调试、监控和做失败恢复。

### 追问点
- 为什么很多 Agent 项目不稳定，本质上是循环设计不清晰？
- 哪一步最容易出问题？

## 题目
什么是 Agent 中的 Planning？为什么很多任务需要先规划再执行？

## 答案
### 核心结论
Planning 指的是 Agent 在执行前先把目标拆分为可操作的子任务或步骤，这样可以降低复杂任务的一次性决策难度，提高成功率和可解释性。

### 关键点
1. 复杂任务往往不是一步能完成的。
2. 先规划能减少遗漏关键步骤的概率。
3. 规划让系统更容易做阶段性检查和失败重试。
4. 有些任务不需要重规划，但多步任务通常从规划中显著受益。

### 工程实践
规划不一定是完整长计划，也可以是短视、滚动更新的局部计划。

### 追问点
- Planning 会不会让链路变慢？
- 哪类任务更适合动态规划而不是静态规划？

## 题目
Agent 里的 memory 是什么？它和普通多轮对话历史有什么区别？

## 答案
### 核心结论
Agent 的 memory 是为了支持任务连续性和决策一致性而保留的状态信息，它不仅包括对话历史，还可能包括中间结果、用户偏好、工具输出和长期知识。

### 关键点
1. 普通对话历史只是时间顺序的消息堆叠。
2. Agent memory 更强调“哪些信息对后续行动有价值”。
3. Memory 可以分为短期会话记忆和长期持久记忆。
4. Memory 的目标不是保存一切，而是保留对任务有帮助的信息。

### 工程实践
好的 memory 设计应关注相关性、时效性和可控写入，而不是无限堆积历史。

### 追问点
- 为什么 memory 过多会让 Agent 变差？
- 长期记忆如何做权限隔离？

## 题目
为什么很多 Agent 架构都要支持工具调用？

## 答案
### 核心结论
因为模型本身擅长语言理解与决策，但不擅长精确计算、实时查询和外部执行；工具调用能把不确定的生成能力和确定性的系统能力结合起来。

### 关键点
1. 工具可以提供实时信息、数据库访问、搜索、代码执行等能力。
2. 工具让 Agent 从“只会说”升级为“能做事”。
3. 工具调用可提升准确率、可审计性和可控性。
4. 工具越强，权限风险也越高，因此架构设计很关键。

### 工程实践
常见原则是：模型负责决策，工具负责执行，系统负责校验和权限控制。

### 追问点
- 工具越多是不是越好？
- 什么能力不适合直接交给模型本身？

## 题目
单 Agent 和多 Agent 架构各自适合什么场景？

## 答案
### 核心结论
单 Agent 更适合逻辑相对集中、链路较短的任务；多 Agent 更适合职责清晰、子任务异构或需要并行协作的大型任务系统。

### 关键点
1. 单 Agent 架构简单，易落地、易调试。
2. 多 Agent 架构可以分工明确，例如规划者、检索者、执行者、审查者。
3. 多 Agent 的好处是职责隔离和扩展性强。
4. 多 Agent 的代价是通信开销、协调复杂度和失败排查难度更高。

### 工程判断
不要为了“看起来高级”而上多 Agent，先确认单 Agent 是否已经足够。

### 追问点
- 多 Agent 最大的收益来自哪里？
- 多 Agent 最大的隐性成本是什么？

## 题目
为什么状态管理是 Agent 架构中的核心问题？

## 答案
### 核心结论
因为 Agent 是连续执行系统，不是单次问答接口；状态决定它知道什么、做到哪一步、下一步该做什么以及是否已经完成目标。

### 关键点
1. 没有状态，Agent 很难做多步任务。
2. 状态包括任务进度、工具结果、中间结论、用户偏好和异常信息。
3. 状态如果混乱，Agent 会重复做事、忘记关键步骤或做出错误决策。
4. 稳定状态设计是实现可恢复、可回放、可调试的基础。

### 工程实践
通常需要明确状态结构、更新时机、持久化策略和过期策略。

### 追问点
- 状态和 memory 有什么区别？
- 哪些状态应该写入持久层？

## 题目
为什么 Agent 需要观察（observation）阶段？不能直接连续执行吗？

## 答案
### 核心结论
观察阶段的作用是读取动作结果并反馈到决策系统中，如果没有 observation，Agent 就无法知道上一步是否成功，也无法形成闭环控制。

### 关键点
1. Agent 决策依赖环境反馈，而不是盲目顺序执行。
2. 工具可能失败、返回空值、超时或返回不完整结果。
3. 只有观察结果，才能决定是否重试、是否改计划、是否结束任务。
4. 闭环控制是 Agent 区别于线性脚本的重要特征之一。

### 工程实践
Observation 不应该只是原始输出，还应该包含结构化状态、错误码和可信度信息。

### 追问点
- 为什么 observation 设计不好会放大幻觉？
- 观察结果是否需要标准化？

## 题目
什么是 Reflection？它在 Agent 架构中起什么作用？

## 答案
### 核心结论
Reflection 指的是 Agent 在执行后回顾自己的结果、错误或计划，并据此进行修正，它的作用是提高复杂任务中的鲁棒性和迭代质量。

### 关键点
1. Reflection 让 Agent 不只是“往前走”，还会“回头看”。
2. 它适合用于代码修复、规划优化、答案审查等任务。
3. Reflection 有助于发现中间步骤错误和不一致。
4. 但 Reflection 也会增加 token 成本和链路长度。

### 工程实践
不是所有任务都需要 reflection，通常在高价值或高失败率任务中更值得引入。

### 追问点
- Reflection 和普通重试有什么区别？
- Reflection 是否会导致系统过度思考？

## 题目
什么是 ReAct 模式？它为什么在早期 Agent 设计中很经典？

## 答案
### 核心结论
ReAct 是一种把“推理（Reasoning）”和“行动（Acting）”交替结合的 Agent 模式，经典之处在于它把多步决策过程显式化，便于调用工具并持续利用环境反馈。

### 关键点
1. Agent 先思考当前问题和下一步动作。
2. 再调用工具或执行动作。
3. 然后读取 observation。
4. 再基于新信息继续思考和行动。
5. 直到得到结果或满足停止条件。

### 优势
- 结构清晰
- 易于扩展工具
- 适合需要逐步探索的任务

### 局限
- 容易链路过长
- 中间思考可能不稳定
- 在高并发场景下成本较高

### 追问点
- ReAct 与 workflow 的根本区别是什么？
- 为什么 ReAct 对工具描述质量敏感？

## 题目
Plan-and-Execute 模式和 ReAct 模式有什么区别？

## 答案
### 核心结论
Plan-and-Execute 强调先生成整体计划，再逐步执行；ReAct 更强调边想边做、逐步迭代。前者结构更稳定，后者更灵活。

### 关键点
1. Plan-and-Execute 通常先产生完整或半完整计划。
2. 然后执行器按计划逐步完成任务。
3. ReAct 则是在每一步都重新思考下一步该做什么。
4. 前者更适合任务边界清晰的多步问题，后者更适合信息不完整、需要探索的问题。

### 工程判断
如果任务流程相对固定，Plan-and-Execute 往往更可控；如果任务依赖动态反馈，ReAct 更自然。

### 追问点
- Plan-and-Execute 是否也需要中途重规划？
- 为什么有些任务不能提前规划完整？

## 题目
什么是 Supervisor 模式？它在多 Agent 系统中解决了什么问题？

## 答案
### 核心结论
Supervisor 模式是由一个上层协调者根据任务状态，把子任务分配给不同 Agent 或工具执行，从而解决多角色协作中的路由、分工和汇总问题。

### 关键点
1. Supervisor 负责全局目标和任务分派。
2. 子 Agent 负责执行具体专业子任务。
3. 这种模式适合复杂工作流和职责明显不同的团队式架构。
4. 它能避免多个 Agent 无序对话和重复劳动。

### 工程风险
Supervisor 成为单点决策中心后，稳定性和决策质量会成为系统瓶颈。

### 追问点
- Supervisor 和 orchestration engine 有什么差别？
- Supervisor 做错路由时怎么兜底？

## 题目
Agent 系统中的路由（routing）层通常负责什么？

## 答案
### 核心结论
Routing 层主要负责根据任务类型、用户意图、上下文状态或风险级别，把请求分配到合适的工具、模型、流程或子 Agent。

### 关键点
1. 路由决定“谁来处理这个问题”。
2. 它可以基于规则、分类模型、LLM 判断或混合策略实现。
3. 好的路由可以节省成本、提升质量和减少无效链路。
4. 路由错误会导致整个后续链路失效或跑偏。

### 工程实践
常见策略是先规则过滤高置信简单场景，再让模型处理模糊和复杂场景。

### 追问点
- 路由为什么经常是 Agent 系统真实瓶颈？
- 路由层是否应该自己可回放、可解释？

## 题目
Workflow 和 Agent 的边界应该怎么理解？

## 答案
### 核心结论
Workflow 更偏确定性编排，步骤和分支通常预先定义；Agent 更偏动态决策，具体行为依赖模型对当前状态的判断。

### 关键点
1. Workflow 的强项是稳定、可预测、易审计。
2. Agent 的强项是灵活、泛化、能处理开放式任务。
3. 两者不是对立关系，很多系统是 workflow + agent 混合架构。
4. 常见做法是用 workflow 控主干，用 agent 解决开放式节点。

### 工程实践
高风险、高确定性链路优先 workflow；探索式、不确定任务可引入 agent。

### 追问点
- 为什么很多所谓 Agent 产品，实际更接近 workflow？
- 什么时候 agent 会退化成不可控 workflow？

## 题目
为什么 Agent 的外部动作需要考虑幂等性？

## 答案
### 核心结论
因为 Agent 可能重试、重复规划或在异常恢复中重复执行同一步骤，如果外部动作不具备幂等性，就可能导致重复下单、重复发消息、重复写数据等严重后果。

### 关键点
1. Agent 失败恢复常伴随重试。
2. 工具调用如果带副作用，必须控制重复执行风险。
3. 幂等键、状态检查和去重机制是常见做法。
4. 幂等性设计是 Agent 从“能跑”到“能上线”的分水岭。

### 工程实践
对支付、工单、审批、邮件发送等操作尤其要做幂等保护。

### 追问点
- 哪些工具天然幂等，哪些不是？
- 为什么读操作也可能需要幂等语义？

## 题目
Agent 系统里做重试时，为什么不能简单“一失败就再来一次”？

## 答案
### 核心结论
因为失败原因可能是权限问题、参数错误、上下文错误或外部系统不可用，盲目重试不但无效，还可能放大成本和副作用。

### 关键点
1. 要先区分可重试错误和不可重试错误。
2. 网络抖动、短时超时通常适合重试。
3. 参数校验失败、权限拒绝通常不应直接重试。
4. 有副作用的操作需要结合幂等机制和人工确认。

### 工程实践
重试策略通常应包含次数限制、退避机制和失败分类。

### 追问点
- 什么情况下应该重规划而不是重试？
- 为什么重试策略要和观测数据联动？

## 题目
为什么复杂 Agent 任务通常需要 checkpoint 或持久化状态？

## 答案
### 核心结论
因为复杂任务可能跨多步、跨系统、跨时间窗口执行，没有 checkpoint 就很难在中断、失败或部署重启后恢复上下文和进度。

### 关键点
1. 长任务不适合全靠内存维持状态。
2. Checkpoint 能记录中间步骤、工具结果和当前任务阶段。
3. 它有助于断点恢复、人工接管和问题回放。
4. 也是支持异步执行和分布式调度的重要基础。

### 工程实践
保存什么、何时保存、恢复到哪一步，都需要明确设计。

### 追问点
- Checkpoint 粒度太细或太粗各有什么问题？
- 哪些信息不应该持久化？

## 题目
为什么很多企业级 Agent 系统都需要 Human-in-the-Loop 设计？

## 答案
### 核心结论
因为 Agent 在高风险场景下不应完全自主执行，人工介入可以作为风险兜底、责任边界和质量校验机制，防止模型错误直接造成业务损失。

### 关键点
1. 高风险动作如审批、发信、删改数据通常应人工确认。
2. 人工介入不仅是安全措施，也是反馈采集机制。

4. 也便于逐步上线：先 workflow，后局部 Agent 化。

### 工程实践
高风险业务通常只在“认知复杂但动作风险可控”的节点使用 Agent。

### 追问点
- 哪些节点最适合 Agent 化？
- 混合架构的最大设计难点是什么？

## 题目
Agent 架构最常见的失败模式有哪些？

## 答案
### 核心结论
常见失败模式包括错误规划、错误路由、工具误用、状态污染、死循环、伪完成和恢复失败，这些问题往往不是单点错误，而是链式叠加结果。

### 关键点
1. 规划错误：一开始就拆错任务。
2. 路由错误：把任务交给不合适的工具或子 Agent。
3. 工具误用：参数错、权限错、语义错。
4. 状态污染：把错误信息写进 memory。
5. 死循环：不断重试或反复讨论同一问题。
6. 伪完成：系统以为任务完成，实际没有。
7. 恢复失败：中断后无法正确续跑。

### 工程实践
失败模式最好做分类统计，这比只看“成功/失败”二分类更有用。

### 追问点
- 哪类失败最难被监控发现？
- 如何建立 failure taxonomy？

## 题目
在 Agent 架构设计里，最核心的 trade-off 之一是什么？

## 答案
### 核心结论
最核心的 trade-off 之一是“灵活性和可控性”的平衡：越自由的 Agent 越能处理开放任务，但也越难预测、验证和治理；越受控的系统越稳定，但适应性越差。

### 关键点
1. 灵活性高意味着模型可以动态规划和决策。
2. 可控性高意味着流程、状态和动作边界都被明确约束。
3. 灵活系统适合探索式任务，可控系统适合高风险业务。
4. 真实落地往往不是二选一，而是按节点做动态取舍。

### 面试表达
可以总结为：Agent 架构不是追求“最聪明”，而是追求“在业务边界内足够聪明、足够稳”。

### 追问点
- 如何把这种 trade-off 产品化？
- 什么情况下应主动牺牲灵活性？

## 题目
如果让你从零设计一个企业级 Agent 平台，你认为最重要的架构能力有哪些？

## 答案
### 核心结论
企业级 Agent 平台最重要的不是单个模型能力，而是围绕模型构建出的任务编排、状态管理、工具治理、权限控制、可观测性和评测闭环能力。

### 关键点
1. 任务编排：支持单 Agent、多 Agent、workflow 混合模式。
2. 状态与记忆：支持短期状态、长期记忆、checkpoint 和恢复。
3. 工具治理：统一 schema、权限、超时、重试和审计。
4. 可观测性：支持 trace、回放、错误分类和成本分析。
5. 安全与权限：最小权限、审批、人工介入、操作留痕。
6. 评测闭环：支持离线评测、线上实验、失败样本回流。

### 工程视角
从平台角度看，Agent 不是一个 prompt，而是一整套运行时系统。

### 追问点
- 平台层和业务层的边界怎么划？
- 如果只能先做三项能力，你会优先做什么？
