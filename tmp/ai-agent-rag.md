## 题目
什么是 RAG？它要解决的大模型核心问题是什么？

## 答案
### 核心结论
RAG 是 Retrieval-Augmented Generation，即“检索增强生成”，它通过在生成前先检索外部知识，再把相关内容提供给模型，从而缓解大模型知识过期、领域知识缺失和幻觉问题。

### 关键点
1. 大模型参数里的知识不是实时更新的。
2. 仅靠模型记忆回答，容易出现幻觉或答非所问。
3. RAG 通过外部知识库给模型提供“当前可依据的信息”。
4. 它特别适合企业知识问答、文档助手、客服和内部搜索场景。

### 面试表达
可以概括为：RAG 让模型从“凭记忆回答”变成“先查资料再回答”。

### 追问点
- RAG 能彻底消除幻觉吗？
- RAG 和微调分别解决什么问题？

## 题目
一个典型的 RAG 系统整体流程通常是什么？

## 答案
### 核心结论
典型 RAG 流程通常包括：文档接入、清洗切分、向量化建库、查询向量化、召回相关片段、必要的重排与压缩、把结果拼进 Prompt，最后由模型生成答案。

### 关键点
1. Offline 阶段：采集文档、清洗、切 chunk、向量化、写入索引。
2. Online 阶段：用户提问后先做查询处理。
3. 然后做召回，可以是向量检索、关键词检索或混合检索。
4. 召回后通常还会重排、裁剪、压缩上下文。
5. 最后把高相关片段交给 LLM 生成答案。

### 工程实践
真正的效果往往不是由“模型多强”决定，而是由“召回质量 + 上下文组织质量”决定。

### 追问点
- 为什么很多 RAG 效果差，问题不在模型而在检索？
- 哪一步最影响准确率？

## 题目
RAG 系统里的“知识库”通常包括哪些数据源？

## 答案
### 核心结论
知识库本质上是可被检索和引用的外部信息集合，常见数据源包括文档、网页、FAQ、产品手册、工单记录、数据库内容和内部 Wiki。

### 关键点
1. 文档类：PDF、Word、Markdown、PPT、网页。
2. 结构化数据：数据库记录、表格、报表、配置项。
3. 半结构化数据：FAQ、帮助中心、工单评论、聊天记录。
4. 内部知识：Wiki、SOP、研发文档、制度说明。
5. 不同数据源清洗方式和切分策略通常不同。

### 工程实践
知识库建设的难点不只是“接进来”，而是如何保证时效性、权限一致性和内容质量。

### 追问点
- 结构化数据是否一定要走 RAG？
- 什么数据不适合直接进知识库？

## 题目
为什么 RAG 系统里需要对文档做 chunk 切分？

## 答案
### 核心结论
因为原始文档通常过长、语义跨度大，直接整体向量化会降低检索粒度；chunk 切分能把知识拆成更适合召回和引用的小单元。

### 关键点
1. 整篇文档向量化会丢失细粒度定位能力。
2. chunk 太大，检索相关性容易被稀释。
3. chunk 太小，语义上下文可能不完整。
4. 合理切分有助于提高召回精度和后续拼接质量。

### 工程实践
chunk 设计不是固定值问题，而是和文档结构、业务任务、模型上下文共同相关。

### 追问点
- chunk 越小是不是越好？
- 为什么同一知识库里不同文档类型可能要不同切分策略？

## 题目
什么是 embedding？它在 RAG 中的作用是什么？

## 答案
### 核心结论
Embedding 是把文本映射成向量表示的方法，在 RAG 中的作用是把查询和文档映射到同一语义空间，从而支持语义相似度检索。

### 关键点
1. 相似语义的文本，在向量空间里通常更接近。
2. 用户问题和知识片段都要被向量化，才能做语义匹配。
3. Embedding 模型决定了召回阶段对语义相似性的理解能力。
4. Embedding 不是生成模型，它主要负责表示和匹配。

### 工程实践
很多 RAG 线上效果差，根因可能是 embedding 模型与业务语料不匹配，而不是 LLM 本身不够强。

### 追问点
- Embedding 模型是不是越大越好？
- 为什么代码库和通用文档可能需要不同 embedding？

## 题目
向量数据库在 RAG 里主要负责什么？

## 答案
### 核心结论
向量数据库主要负责存储文档向量、建立相似度索引，并在查询时快速返回与用户问题最相关的候选片段。

### 关键点
1. 它不是只存文本，而是存文本对应的向量和元数据。
2. 常见能力包括相似度搜索、过滤、分页、分区、近似最近邻搜索。
3. 元数据常用于权限过滤、时间过滤、文档类型过滤。
4. 真正可用的 RAG 往往依赖“向量检索 + 元数据过滤”组合能力。

### 工程实践
向量数据库选型时，不只是看检索速度，还要看过滤能力、可扩展性和运维成本。

### 追问点
- 向量数据库和普通数据库的角色区别是什么？
- 为什么只做向量检索往往不够？

## 题目
RAG 中 top-k 是什么？top-k 越大效果一定越好吗？

## 答案
### 核心结论
Top-k 指召回时返回前 k 个最相关片段；它不是越大越好，因为更多片段会提高覆盖率，但也会带来噪声、成本和上下文污染。

### 关键点
1. top-k 太小，容易漏召回关键信息。
2. top-k 太大，容易引入低相关片段。
3. 低相关内容进入上下文后，可能干扰模型回答。
4. top-k 需要结合 rerank、上下文预算和业务评测调优。

### 工程实践
实际系统里经常先召回较大 top-k，再用 rerank 压缩成更小的最终上下文集。

### 追问点
- top-k 应该离线调还是在线调？
- 为什么 top-k 和 chunk 大小是联动关系？

## 题目
关键词检索和向量检索有什么区别？

## 答案
### 核心结论
关键词检索更依赖字面匹配，向量检索更依赖语义相似；前者擅长精确词命中，后者擅长同义表达和语义泛化。

### 关键点
1. 关键词检索适合专有名词、编号、术语、精确字段。
2. 向量检索适合同义改写、自然语言问题和语义模糊表达。
3. 关键词检索可解释性强，向量检索泛化能力更好。
4. 实际业务中，两者往往结合使用，而不是二选一。

### 工程实践
如果用户经常问型号、工单号、产品名，纯向量检索通常不够稳。

### 追问点
- 为什么关键词检索在企业场景里依然重要？
- 什么时候向量检索会明显优于 BM25？

## 题目
什么是 hybrid search？为什么很多 RAG 系统最后都会走向混合检索？

## 答案
### 核心结论
Hybrid search 是把关键词检索和向量检索结合起来，以同时兼顾字面精确命中和语义召回能力，因此在真实业务里通常比单一路径更稳。

### 关键点
1. 关键词检索擅长精确词和稀有实体。
2. 向量检索擅长语义相似和自然语言表达。
3. 混合检索可以减少“关键词漏掉”和“语义找偏”的双重问题。
4. 很多企业场景中，混合检索比纯向量检索更符合真实输入分布。

### 工程实践
常见方案是并行召回两路结果，再做合并、去重和重排。

### 追问点
- 混合检索里的分数如何融合？
- 哪些场景其实没必要上 hybrid？

## 题目
什么是 rerank？为什么很多 RAG 系统需要在召回后做重排？

## 答案
### 核心结论
Rerank 是对初步召回的候选片段再次做更细致的相关性排序，它能提升最终送入模型的上下文质量，因此对 RAG 精度影响很大。

### 关键点
1. 初召回通常偏快、偏粗糙。
2. Rerank 模型更适合对“问题和候选片段是否真正相关”做精排。
3. 它能在有限上下文预算下保留更高价值片段。
4. 没有 rerank 时，低相关内容更容易混入 Prompt。

### 工程实践
常见做法是先召回 20 到 50 条，再重排取前 3 到 8 条进入模型。

### 追问点
- 为什么 rerank 往往比继续扩大 top-k 更有效？
- rerank 会带来哪些成本？

## 题目
如何设计合理的 chunk 切分策略？需要考虑哪些因素？

## 答案
### 核心结论
合理的 chunk 策略要在“语义完整性、检索粒度、上下文长度和索引成本”之间平衡，不能只看固定字符数或 token 数。

### 关键点
1. 文档结构：标题、段落、列表、表格、代码块是否应整体保留。
2. 任务类型：问答、摘要、抽取对粒度要求不同。
3. 模型上下文：最终能放进 Prompt 的 token 数有限。
4. 检索策略：是否有 rerank、是否做句级或段级召回。
5. 文档类型：政策文档、FAQ、代码文档的切分逻辑不同。

### 工程实践
优先考虑基于结构和语义边界切分，而不是纯字符截断。

### 追问点
- 什么情况下应该用滑窗切分？
- 表格和代码块应如何切分？

## 题目
为什么 chunk 切分时常常会设置 overlap（重叠）？

## 答案
### 核心结论
Overlap 的作用是缓解语义跨边界被切断的问题，让相邻 chunk 共享部分上下文，从而提高召回时的完整性和鲁棒性。

### 关键点
1. 文档重要信息常常跨句子、跨段落分布。
2. 没有 overlap 时，关键语义可能被切在边界两侧。
3. 适当重叠能提升召回完整性。
4. 但 overlap 太大会增加索引体积、重复召回和噪声。

### 工程实践
Overlap 的大小通常要结合 chunk 长度、文档类型和重复控制策略一起调。

### 追问点
- overlap 多大比较合适？
- overlap 会不会导致模型看到大量重复内容？

## 题目
为什么元数据过滤在企业级 RAG 里非常重要？

## 答案
### 核心结论
因为企业知识库不仅要“找得到”，还必须“只找该用户有权限看的内容”；元数据过滤是实现权限控制、范围控制和召回精度提升的关键机制。

### 关键点
1. 元数据可包含部门、文档类型、创建时间、语言、权限范围等。
2. 检索前或检索时过滤能减少无关候选。
3. 也可以防止低权限用户召回高权限内容。
4. 企业场景里，权限错召回往往比普通答错更严重。

### 工程实践
不要把权限过滤放在生成后再补救，最好在检索阶段就做硬过滤。

### 追问点
- 元数据过滤和 Prompt 里的权限提醒有什么不同？
- 检索后过滤会带来什么风险？

## 题目
什么是 query rewrite？它在 RAG 中通常起什么作用？

## 答案
### 核心结论
Query rewrite 是把用户原始问题改写成更适合检索的表达形式，它的作用是提高召回命中率，尤其适用于用户表达口语化、模糊化或省略上下文的场景。

### 关键点
1. 用户原问题不一定适合直接检索。
2. 改写可以补足缺失实体、规范术语、展开指代。
3. 也可以从问句改成关键词式或检索式表达。
4. 错误改写也可能引入召回偏差，因此需要评估。

### 工程实践
多轮对话里，query rewrite 往往先要做上下文补全，再做检索改写。

### 追问点
- query rewrite 和用户可见改写是否应该一致？
- 改写失败会对召回造成什么连锁影响？

## 题目
什么是 multi-query retrieval？它适合什么场景？

## 答案
### 核心结论
Multi-query retrieval 是为同一个问题生成多个不同表述的查询，再分别召回并合并结果，从而提高召回覆盖率，适合表达多样、术语别名多、长尾问法丰富的场景。

### 关键点
1. 同一问题可以有多种问法。
2. 单一查询可能只命中知识库的一部分表达方式。
3. 多查询可以覆盖不同关键词、同义词和角度。
4. 代价是检索成本更高、结果去重和重排更重要。

### 工程实践
适合高价值搜索、复杂问答，但未必适合所有实时低延迟场景。

### 追问点
- multi-query 和 query rewrite 的关系是什么？
- 多查询会不会放大噪声召回？

## 题目
RAG 里为什么“召回到了”不等于“模型就能答对”？

## 答案
### 核心结论
因为召回只是拿到了候选资料，能否答对还取决于上下文拼接方式、片段排序、冗余控制、Prompt 约束以及模型是否正确引用这些内容。

### 关键点
1. 召回命中不代表模型一定关注到关键片段。
2. 低质量排序会让重要内容埋在后面。
3. 冗余或矛盾片段会干扰模型判断。
4. Prompt 如果没有要求“基于资料回答”，模型仍可能凭常识扩展。
5. 引用方式不清晰也会降低利用率。

### 工程实践
RAG 的后半段其实是上下文工程问题，不只是检索问题。

### 追问点
- 片段排序是否会显著影响答案质量？
- 为什么有时减少上下文反而效果更好？

## 题目
为什么很多 RAG 系统会要求模型返回引用来源？

### 工程实践

## 题目
如果知识库里没有答案，RAG 系统应该怎么处理？

## 答案
### 核心结论
当知识库中没有足够依据时，系统应优先稳健拒答或请求补充信息，而不是让模型强行编造；无答案处理能力是 RAG 可靠性的关键之一。

### 关键点
1. 无答案场景比有答案场景更能检验系统真实性。
2. 应设置低置信召回、低相关结果时的拒答策略。
3. Prompt 中要明确“无依据不猜”。
4. 必要时可降级为搜索、人工转接或澄清提问。

### 工程实践
很多线上事故不是“答错简单题”，而是“没依据时还编得很像”。

### 追问点
- 如何判断“没答案”而不是“没召回到”？
- 拒答率和用户体验如何平衡？

4. 知识更新机制直接影响用户是否信任系统。

### 工程实践

- 多版本文档共存时如何避免冲突召回？

## 题目

## 答案
### 核心结论
召回率强调“该找到的有没有找到”，准确率强调“找到的内容里有多少是真的相关”；RAG 系统优化时经常要在两者之间做平衡。

1. 召回率低会导致关键证据根本没进入系统。

3. 高召回不代表高可用，因为噪声多也会拖垮生成质量。
4. RAG 的目标通常不是单独最大化某个指标，而是最大化最终答案质量。

### 工程实践
召回阶段可偏高召回，后续再通过 rerank 和压缩提升最终有效精度。

### 追问点
- 为什么生成质量有时和召回指标并不线性对应？

## 题目

## 答案

## 题目
RAG 系统为什么特别容易受到来自文档内容的 prompt injection 影响？

## 答案
### 核心结论
因为 RAG 会把外部文档内容直接拼进模型上下文，而模型无法天然区分“这是一段知识”还是“这是一段恶意指令”，所以文档本身可能变成攻击载体。

### 关键点
1. 恶意内容可能出现在网页、论坛、邮件、共享文档中。
2. 一旦被召回，就可能影响模型行为。
3. 如果 RAG 又连接工具或 Agent，风险会进一步扩大。
4. 仅靠 system prompt 说“忽略文档中的指令”并不总可靠。

### 工程实践
需要结合内容清洗、来源白名单、隔离渲染、工具权限控制和高风险动作确认机制。

### 追问点
- 文档注入和用户输入注入有何不同？
- 哪些 RAG 场景风险最高？

## 题目
如果让你设计一个企业内部知识助手的 RAG 架构，你会重点考虑哪些权衡？

## 答案
### 核心结论
核心权衡通常是效果、延迟、成本、安全和知识新鲜度之间的平衡：召回更广可能更全，但也更慢更贵；权限更严格更安全，但实现更复杂。

### 关键点
1. 检索精度和召回覆盖之间的平衡。
2. 上下文长度和成本之间的平衡。
3. 缓存收益和知识实时性之间的平衡。
4. 多租户权限隔离和系统复杂度之间的平衡。
5. 引用完整性和用户阅读负担之间的平衡。

### 工程实践
成熟系统往往不是“能力拉满”，而是“在业务目标下做最合适的权衡”。

## 题目
如果线上用户反馈“RAG 经常答非所问”，你会如何系统排查？

## 答案
### 核心结论
应沿着“查询理解、召回质量、重排质量、上下文拼接、Prompt 约束、最终生成”这条链路逐层排查，而不是只盯着模型输出本身。

### 关键点
1. 看用户原问题是否需要改写和补全。
2. 看召回结果是否真的包含正确证据。
3. 看 rerank 是否把正确证据排在前面。
4. 看上下文是否被噪声或重复内容污染。
5. 看 Prompt 是否要求基于证据回答。
6. 看模型最终是否正确引用和整合证据。

### 工程实践
保留每次请求的 trace，包括 query、召回片段、排序结果、最终 Prompt 和答案，是排障前提。

### 追问点
- 如何快速区分“没召回”和“没用好召回”？
- 哪类日志最值得长期采集？

## 题目
你认为 RAG 系统设计里最核心的 trade-off 是什么？

## 答案
### 核心结论
最核心的 trade-off 是“召回覆盖、上下文纯度、成本和时延”之间的平衡：想回答得更全面，往往要召回更多；召回更多又会带来噪声、更长上下文、更高成本和更慢响应。

### 关键点
1. 高覆盖有助于减少漏答，但容易引入噪声。
2. 高纯度有助于生成稳定，但可能漏掉关键证据。
3. 更复杂的检索和重排通常效果更好，但也更慢更贵。
4. 企业场景里还要额外平衡权限控制和知识更新复杂度。

### 面试表达
可以总结为：RAG 不是“把资料塞给模型”这么简单，而是持续在“找得全、找得准、喂得好、跑得动”之间做工程权衡。

### 追问点
- 哪类业务更优先考虑高纯度而不是高覆盖？
- 如何通过路由、rerank 和压缩缓解这些 trade-off？
